把模糊需求转成可执行对象、确保关键知识在运行时可达、编码机械性规则防止模式漂移……Harness Engineering 是一套为 AI Agent 设计约束、引导和验证机制的系统工程方法。

本文将 Harness Engineering 的实践方法整理为 9 大模块，每个模块拆解为具体的方法、描述和产出物，作为工程落地的参考清单。

## 01 任务组织层

把模糊需求转成可执行对象，而不是直接一句"帮我做个功能"。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 1.1 | 需求结构化拆解 | 将需求拆解为：目标、范围、验收标准、约束条件、相关文件、禁改区、输出要求 | 结构化任务卡 |
| 1.2 | Initializer Agent 预处理 | 用 Initializer Agent 读取需求，扩展为详细 feature 列表，标记初始状态 FAILING | feature_checklist.md |
| 1.3 | 任务粒度控制 | 每个任务限定为"单次会话可完成"的粒度，避免 agent 一次想做太多 | 可执行子任务队列 |
| 1.4 | 禁改区声明 | 在任务描述中明确标注哪些文件/目录/模式不允许修改 | 任务约束清单 |

## 02 知识与上下文层

解决"Agent 看什么"——确保关键知识在运行时可达。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 2.1 | AGENTS.md 作为索引 | 将 AGENTS.md 作为 docs/ 的结构化目录，每条规则拆成独立文档 | AGENTS.md + docs/ |
| 2.2 | 知识资产版本化 | 将 specs、plans、schema、ADR 等沉淀为仓库内可版本化资产 | docs/specs/、docs/adr/ |
| 2.3 | 渐进式上下文注入 | 根据任务阶段动态注入相关上下文，而非一次性灌入全部信息 | 上下文组装策略 |
| 2.4 | 代码符号搜索 | 为 Agent 配置代码搜索工具，按符号、文件名、依赖关系定位代码 | 搜索工具配置 |
| 2.5 | 隐性知识显性化 | 将散落在飞书、企微、口头经验、脑子里的设计决策沉淀为 Agent 可读文档 | decision records |

## 03 系统提示与黄金原则

编码到仓库中的机械性规则，防止模式漂移。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 3.1 | 黄金原则编码 | 将架构约束、代码风格、命名规范等核心原则写入仓库作为硬约束 | docs/golden-rules.md |
| 3.2 | 原则机械化 | 把"请遵守架构原则"变成 lint 规则、CI 检查、自动化验证 | lint 配置 / CI pipeline |
| 3.3 | 短入口 + 结构化知识库 | 系统提示保持精简，指向结构化知识库，不塞进超长 instruction | 精简 system prompt |
| 3.4 | 仓库熵增治理 | 定期用清理 Agent 做 repo hygiene，清过时规则、冗余文档、AI 垃圾 | 清理流程 / 清理 Agent |

## 04 工具与 MCP Servers

Agent 与外部世界交互的接口。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 4.1 | 工具清单最小化 | 工具太多诱导乱调用，太少做不成事。保持最小但够用 | 工具清单 |
| 4.2 | 面向 Agent 的工具设计 | 工具输入输出格式、错误信息要对 Agent 友好 | Agent-friendly 工具 |
| 4.3 | MCP Server 接入 | 通过 MCP 协议将文件读写、代码执行、测试运行等能力暴露给 Agent | MCP 配置 |
| 4.4 | 人工介入节点 | 在关键决策点设置 HITL 节点，Agent 不确定时主动请求人工审核 | HITL 配置 |

## 05 控制流层

定义 Agent 按什么顺序做事。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 5.1 | 标准执行流程 | 读需求 → 搜索代码 → 输出计划 → 小步修改 → 跑检查 → 失败修复 → 门禁通过 | 执行流程模板 |
| 5.2 | Initializer + Coding Agent 双结构 | Initializer 完成环境搭建后退出，Coding Agent 接管后续每轮会话 | 双 Agent 调度配置 |
| 5.3 | Generator + Evaluator 对抗 | Generator 生成，Evaluator 按质量、原创性、可用性、一致性打分反馈 | GAN 风格 Agent 编排 |
| 5.4 | 多 Agent 编排链 | initializer → planner → generator → evaluator 各司其职 | Agent 编排配置 |
| 5.5 | 小步迭代约束 | 限制单次修改步长（只改一个函数/一个文件），降低回滚成本 | 步长约束规则 |

## 06 验证与反馈层

对 Agent 输出做系统性验证，构建闭环。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 6.1 | 分级验证管道 | 按风险等级分层：语法检查 → 测试执行 → 语义审查（LLM Review） | 验证管道配置 |
| 6.2 | 自动测试执行 | 每次修改后自动运行单测、集成测试、e2e、类型检查、lint | CI / 测试脚本 |
| 6.3 | LLM 语义审查 | 对高风险输出用另一个 LLM 做语义审查，检查逻辑和架构一致性 | LLM Reviewer 配置 |
| 6.4 | 失败反馈回注 | 验证失败时将具体错误信息回注给 Agent，驱动修正 | 反馈模板 |
| 6.5 | 生产遥测二次确认 | 代码通过后用生产遥测（延迟、错误率、资源消耗）做二次确认 | 遥测告警规则 |

## 07 Hooks 与反压机制

在关键动作前后注入检查，控制风险。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 7.1 | Token 预算上限 | 为每个任务设置 token 消耗上限，避免失控产生异常账单 | 预算配置 |
| 7.2 | 会话时长上限 | 为每次会话设置最大执行时长，超时保存状态并退出 | 超时配置 |
| 7.3 | 动作前置检查 | 在文件写入、代码提交、外部 API 调用等关键动作前注入检查 | pre-action hooks |
| 7.4 | 安全门禁 | 对修改生产配置、删除文件、访问敏感数据等高风险操作设置审批 | 安全策略配置 |
| 7.5 | 限速与降级 | Agent 进入高频重试循环时自动限速或降级到保守策略 | 反压策略 |

## 08 持久化与跨会话记忆

解决长任务中的"失忆"问题。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 8.1 | Feature Checklist 驱动 | 用 checklist 跟踪每个 feature 的 FAILING / PASSING 状态 | feature_checklist.md |
| 8.2 | Session Handoff 文件 | 每次会话结束前重写交接文件，记录进度、环境状态、下一步计划 | session_handoff.md |
| 8.3 | 会话结束清理与交接 | 清理临时文件、提交中间成果、更新 checklist、重写 handoff | 交接 artifacts |
| 8.4 | 状态恢复优先 | 每次新会话先读取 handoff 和 checklist 恢复上下文再开始工作 | 启动流程 |

## 09 可观测性与遥测

持续优化的基础。

| # | 方法 | 做什么 | 产出物 |
|---|------|--------|--------|
| 9.1 | 工具调用追踪 | 记录每次工具调用的输入、输出、耗时和 token 消耗 | trace 日志 |
| 9.2 | Failure Mode Analysis | 通过 traces 找高频失败模式，针对性改进 harness | 失败模式报告 |
| 9.3 | 评估管道 | 定义客观成功标准（通过率、代码质量分、回归率），做 A/B 测试 | eval pipeline |
| 9.4 | 生产遥测反哺 | 将延迟、错误率、资源消耗等生产数据纳入反馈闭环 | 反馈闭环配置 |
| 9.5 | 错误分布与行为轨迹 | 分析 Agent 行为轨迹，发现非预期模式（无效重试、死循环、路径偏离） | 行为分析仪表盘 |
